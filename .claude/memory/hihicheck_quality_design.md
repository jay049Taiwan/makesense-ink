## What
target DB（DB01–05、DB07–09，共 8 個）新增 4 個品管欄位，讓家族 pipeline 的品質進步可量化：

- `字數目標`（number）：企劃必填、四九講了用四九的、沒講用文類通用建議
- `退回次數`（formula）：數 `執行構想` 內「四九審查意見」這 6 個字出現次數
- `過期`（formula）：`now() > 截止時間`（DB07/08/09 無截止時間欄、跳過此 formula）
- `每千字退回次數`（formula）：退回次數 × 1000 ÷ 字數目標

**進步 KPI = 每千字退回次數逐月下降**。同樣字數能少退、或同樣退回數能吃更長內容，都代表家族在學會。

## Why
原本檢核設計要對成品 4 維度 1–10 打分（合構想度／字數達標／真實度／完整度），四九否決：「不通過就退回，分數無增量資訊」。改成「失敗記錄器」框架——兩 channel：
- **人工失敗**：四九修正意見 → 嗨嗨總管 append 進 target `執行構想`，固定 marker `── YYYY-MM-DD 四九審查意見 ──`（這是循環優化第四層機制 owner）
- **系統失敗**：formula 自動算（過期、缺項等）

檢核當判讀器、不發明第三個分類欄位、不打分。萃取庫從這兩 channel 吃負向訊號（總管的封存萃取模式做）。

## How to apply
- target page 看 `每千字退回次數` + `過期` 燈號評估該篇進度健康度
- marker 格式 `── YYYY-MM-DD 四九審查意見 ──` 必須一字不變（formula 數「四九審查意見」6 字）
- DB05 字數目標 fallback 對照表寫在 4-2-2 嗨嗨企劃工作指示 §十（地方通訊月/季/年實證、其餘推估）
- 檢核三鐵律寫在 4-2-7 嗨嗨檢核工作指示頁 §七

## Notion formula 怪坑
`退回次數` formula 因 Notion 型別推論 quirk，**被別的 formula 引用會炸 type error**（試 `prop("退回次數") + 0` 都報錯）。`每千字退回次數` 無法寫成 `退回次數 / 字數目標`，必須 **inline 重算** marker count 的整段運算式。代價：marker 常數「四九審查意見」出現在兩條 formula 裡，未來改 marker 格式必須兩處同步改。

## ⚠️ 更新（2026-05-22）：本 KPI 設計已棄用，改用「被引總數」參考傳播

> 上面整段（字數目標 + 退回次數 / 過期 / 每千字退回次數）為歷史脈絡，**已不再是現況**。下個 session 不要把退回次數 KPI 當有效機制、也不要重建。

### 棄用什麼、為什麼
- 四九 拍板：數「四九審查意見」marker 出現次數（bounce-count）**不是有用信號**——退回次數低不代表東西好，且 Notion formula 型別 quirk（見上「Notion formula 怪坑」）讓它脆、難維護。
- 退回次數 / 每千字退回次數 / 過期 等殭屍欄位**四九自己清**（跨 8 DB 刪欄位，AI 不動）。

### 改用什麼：被引總數驅動的參考傳播閉環
- **新 formula `被引總數`**（已加到 8 個 target DB：DB01-05、DB07-09）= 9 個「被引」relation（提案被引/管考被引/項目被引/協作被引/內容被引/明細被引/庫存被引/對象被引/日期被引）筆數加總。
  - 跟退回次數不同：數的是 relation 筆數（穩、不會炸 type error），不是文字 marker。
  - 被引總數高 = 被家族一再回頭參考 = 經實證的好範本。由聯想平常寫 X引用 →（Notion dual-sync 自動鏡射）成 X被引 自然累積，**不靠主觀點名**。
  - 注意：現在多半是 0，會隨聯想累積引用而長出來（有機累積，非壞掉）。
- **hihigm 對標排序改了**（`.claude/skills/hihigm/SKILL.md`，已 commit）：總管連續問答找對標，從「挑最近完成」改為：
  1. 主排序 = `被引總數` 由高到低（8 DB 都有此欄位）
  2. 通過審查者優先：該 DB 若有「檢核狀態」欄位，只取其「完成/綠燈」群組值的 page。**各 DB 綠燈值不一，子 agent 讀該 DB schema 動態判定、不寫死**：DB03/DB05=`確認ok`、DB02=`已檢核`、DB01=`已封存`、DB09=`完成`；**DB04/07/08 根本沒有「檢核狀態」欄位** → 退用「發佈狀態=已發佈」當通過代理，再無則略過此門檻
  3. tiebreak = 最近完成日期
- **「黃金範例」定義** = 通過審查 + 被引總數高（落地了 SKILL future-note「黃金範例 > 已發佈 > 最近」那句）。
