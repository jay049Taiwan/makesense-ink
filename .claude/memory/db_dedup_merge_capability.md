## What
> **2026/05/21 更新（四九拍板）**：本能力歸屬由嗨嗨總管改為**嗨嗨檢核**。理由：巡查職能已於 2026/05/19 移交檢核，重複偵測/合併執行隨巡查一起歸檢核；檢核本就是 DB08 活躍度掃描 owner、有 code 執行＋派子 agent 能力。分工：嗨嗨分析只「標記」疑似重複（情況B 輸出清單）、不自己合併；嗨嗨檢核負責合併執行 WF（見 hihianly_manual_workflow.md 疑似重複段）。

建一個**常駐能力**（非一次性），掛在**嗨嗨檢核**的巡查/稽核職責下，定期偵測並（經四九審批後）合併 DB01-09 的重複/相近資料。

**兩層架構，中間隔四九審批**：
- A 偵測掃描層：跨 DB01-09 掃 dup/near-dup，只產候選不寫入。字串級（台/臺、編輯距離）走 code node，語意級（英文↔中文、簡稱判斷）需 Claude 節點。建議擴充現有 nvc，不另起爐灶
- B 合併執行層（n8n WF，通用合併執行器）：只吃四九 approved 的群組，union 合併（relation/multi-select/rich text/page content 全保留）+ 單值衝突 Telegram 關卡 + 封存被併頁 + dry-run + idempotent + 操作 log。DB-agnostic，帶「per-DB 合併設定表」

**關鍵原則**：整併不可丟資料；存活頁=建立時間最早；被併頁封存不硬刪；title 不算衝突（存活頁 title 留著、被併頁 title 進別名欄）；merge 永遠過四九審批；「相近」語意因 DB 而異（DB08 多半該併、DB01 同案不同年度標不該併、DB05 同活動多筆報名不該併）。

**DB08 = 第一次實跑**：9,723 筆。已 nvc 掃出：重複 title 17 組、台/臺 17 對（四九已 approved，台灣組已手動合併完成不重做）、純英文 454 筆、簡稱組約 50 組、581 錯置（另案、改類型/封存、不走合併執行器）。

## Why
四九原訂「DB08 每筆建 6 個 DB06 棒次頁」（≈5.8 萬筆），動工前發現 DB08 資料髒，決定先清理。手動逐頁不可行 → n8n WF。且不該是一次性，要升級成常駐掃 DB01-09 的能力。
