目標：DB07 每筆商品由嗨嗨家族六棒跑完，產出正確的 `簡介摘要`（300-500 字），供官網上架販售。

**Why:** DB07 約 4,000+ 筆（99% 是書），`簡介摘要` 3,400+ 筆空白；商品頁多半空白、欄位（登記作者/發行/頁數）也空，內部資料撐不起 300-500 字。改由嗨嗨搜查上網採集 → 家族六棒處理 → 文案產出。

**How to apply:**
- 一筆 DB07 = 6 個 DB06「細部流程」工項：明細類型=細部流程、ai模式=企劃/搜查/分析/聯想/文案/檢核、排序 1-6（企劃1搜查2分析3聯想4文案5檢核6）、ai狀態=待執行、`對應庫存`→該 DB07 page。title 格式 `{品名}｜{棒名}`。
- 範圍：先做「發佈狀態=待發佈 或 已發佈」共 240 筆（1,440 個 DB06 page）。其餘 3,700+ 筆「無發佈」之後再分批。
- create-pages 的 relation 欄位要傳完整 notion.so URL，不能傳裸 UUID。

**關鍵決定（2026-05-18 四九 明示）：**
- 240 筆大批量跑時**改無關卡自動跑**（不要 reviewer-in-the-loop 的 Telegram 關卡 A/B），六棒自動接力，跑完再一次驗收。理由：240×6×2 ≈ 2,880 次 Telegram 確認不可行。
- 簡介摘要與下游棒**必須使用嗨嗨搜查為該筆 target 採集進 DB06 的資訊**。
- DB07 的 240 筆名單相當乾淨，只有 1 組真重複（「行進！鐵支路2024日治臺灣鐵道寫真」vs 無驚嘆號版）；240 筆全建工項、重複之後再處理。
- 240 筆做完後，接著做全 DB07 4,000+ 筆的深度重複稽核。

**n8n 現況與待辦：**
- 現有 Kickoff `K7gzlihO43rFJc59`（DB05 Kickoff v3）的 Query DB06 只用 `對應內容`（→DB05）撈工項，撈不到 DB07 工項（DB07 工項用 `對應庫存` 掛）。
- 待建：DB07 用的「無關卡批次自動跑」workflow（取代逐筆 + 關卡）。
- 指南矛盾待四九修：官網維護指南 §4.1 寫 products.description ← DB07「產品介紹」，但 DB07 實際欄位是 `簡介摘要`（schema 註記 Supabase）；同份指南又把簡介摘要列「不需同步」。以實際 schema 為準（欄位名=簡介摘要）。
