> 用詞約定：本檔不用「棒」這種簡稱。一律寫全稱「DB06 分析記錄(DB06 analysis record)」，指 ai模式=分析 的那筆 DB06清單明細頁。英文詞後面一律括號中文。

## What
以後常有資料要分析（例：成果報告書一份一份來）。手動跑嗨嗨分析時，一律照下面三段走，不再抄捷徑直接寫在 target(目標頁) 上。

### 觸發
每份待分析的 target(目標頁)（DB05 等）先在它的 `對應明細` 欄掛一筆 **DB06 分析記錄**：
- `明細名稱` = 「分析｜<target 名>」
- `ai模式` = 分析
- `ai狀態` = 執行中
- `對應內容` → target(目標頁)（DB06→DB05 relation(關聯欄位)；會鏡射回 target(目標頁) 的 `對應明細`）
- `數據選項` = 內部數據 / 外部數據（自家報告通常內部）

### 執行
1. 讀 target(目標頁) 全文（大頁會被 fetch(抓取) 存檔，用 python 切片讀完 100%，圖片 URL(網址) 濾掉）
2. 抽 5 類具名實體（人物 / 地點 / 單位 / 事件 / 議題）——**全抽全建、不判斷值不值得**；只有真通名（沒名字的咖啡店/市集）、描述型（老廟/街口）不建
3. 對 DB08 **語意查重**（title(標題) + 同義備註，正規化後比對；羅東/羅東鎮、書店/書房 視為同一）
4. target(目標頁) 的 `對應標籤` 連既有 DB08 或新建（新建：`經營類型=紀錄`、`簡介摘要`留空給文案、`同義備註`只從內文抽、換行分隔）。**只補不刪**
5. target(目標頁) 寫 `分析備註`＝ 3-token(語素) 範本 `[類型]｜[執行時間]｜[關聯 DB]`（外層全形｜、內層半形/、每 token(語素) 方括號；格式 single source(單一來源) = 欄位字典 §1.D）

### 收尾
- DB06 分析記錄 寫 `執行備註`（工作對象 / 工作料 / 做了什麼 / 疑似重複 / 未做清單）
- DB06 分析記錄 `ai狀態` = 完成
- **執行備註寫在 DB06 分析記錄（觸發頁），不寫在 target(目標頁)**；target(目標頁) 只留 `分析備註` + `對應標籤`（target(目標頁) 的 `執行備註` 最多留一行指標指向 DB06 分析記錄）

### 疑似重複
DB08 既有同一實體多筆（情況 B）→ **只標記輸出、不自己合併/刪**，待嗨嗨檢核的 merge workflow(合併流程) 處理（見 db_dedup_merge_capability.md）。若四九當場指定留哪筆，先各取一筆連入並在 DB06 分析記錄 標記待合併。

**輸出格式（2026/05/21 四九定，以後都這樣）**：分析發現重複就整理成「可直接貼給檢核 session(對話階段) 的合併清單」——每組列：各 page(頁面) 的 title(標題) + 完整 notion.so URL(網址) + 建議存活頁 + 一句理由。四九 把清單轉貼到另一個（檢核）session(對話階段) 去執行合併。分析這端只列清單、絕不自己動合併/刪除。原因：分析沒有刪除/封存頁面的工具（只有 create/update/move），且合併的 inbound relation(入向關聯) 轉指＋刪除屬危險操作，歸檢核 n8n WF(n8n 工作流)。

## Why
- 把流程導回正規：工作指示的觸發機制本來就是「DB06 分析記錄（ai模式=分析 + ai狀態=執行中）→ 做完 → ai狀態=完成」，DB06 分析記錄 平常由四九/n8n 建，手動跑時改由分析自己建。
- 有 DB06 分析記錄 才有可追溯的工作紀錄落點（執行備註），下一階段/檢核接得起來。

## How to apply
四九 丟一份要分析的資料（或說「一份一份來」）時，直接照上面跑，不用再問流程。

## 關鍵 ID(編號) / 格式
- DB05 collection(資料源)（登記內容）= `28a667a9-ede1-466a-9f18-419da33a8810`
- DB06 collection(資料源)（清單明細，DB06 分析記錄 建這裡）= `a809ff25-fdab-8236-b491-87496d236ac9`
- DB08 collection(資料源)（關係對象，標籤建這裡）= `6934a808-b79b-4446-98dd-f699476408a0`
- relation(關聯欄位) 寫入格式 = **page(頁面) URL(網址) 的 JSON array(JSON 陣列) 字串**（Notion MCP update/create-page 的 SQLite TEXT(SQLite 文字欄)）
- DB06 欄位：`明細名稱`(title 標題) / `ai模式` one of [搜查,分析,聯想,企劃,文案,檢核] / `ai狀態` one of [待執行,執行中,完成] / `對應內容`→DB05

## 首例
2026-05-21 分析 DB05「2015-11-26 104實體書店成果報告書Stay旅人書店」(`2839ff25fdab814cbb85ca4ae1fb6fd2`)。DB06 分析記錄 = `3679ff25fdab813999a5c19f38ff96ba`。target(目標頁) 補 26 對應標籤（新建 8 DB08）、寫 3-token(語素) 分析備註。

## 補充：分析憑證 + 掃描順序（2026-05-22 四九定，往後一律照此）

### 結構性憑證（鐵律）
每分析一個 page(頁面)，憑證落在 **DB06 分析記錄 的兩個結構性條件**，不只靠 target(目標頁) 的 `分析備註` token(語素)：
1. **掛 relation(關聯欄位)**：來源 page(頁面) 的 `對應明細` 必須掛一筆 DB06 分析記錄，欄位組合 = 分析（`ai模式=分析`）。
2. **執行狀態=完成**：收尾時把那筆 DB06 分析記錄 的 **`執行狀態` 設為「完成」**（注意：這是 `執行狀態` 欄，跟既有的 `ai狀態=完成` 是兩個不同欄位；兩個都要設）。
> 用意：`對應明細` relation(關聯欄位) + `執行狀態=完成` 是可查詢、可跑報表的結構性憑證，比純文字 `分析備註` token(語素) 更可靠，能直接掃出「哪些 page(頁面) 已被分析過 / 還沒」。

### 掃描順序（逐類掃描）
DB05 分類別逐一掃完，順序：**成果報告 → 提案申請 → 其餘類別逐一**。每個類別用 dbanly 掃出對應標籤為空的 page(頁面)，一批批清。

### 補建範圍
**此規則由 2026-05-22 起往後套用**；本 session(對話階段)（及更早）已建、未掛 `對應明細`／`執行狀態` 還是執行中的舊 DB06 分析記錄 **不回頭補建**。
