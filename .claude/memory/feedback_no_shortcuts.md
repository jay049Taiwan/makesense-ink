**核心問題**：Claude 訓練優化「快、helpful、不冒犯」，這跟 四九 要的「徹底、嚴格、不偷工」會打架。長對話 token 壓力下，Claude 會越走越傾向「差不多就好」、「下個 session 接力」、「先空著」。

**四九 設計關卡的用意**：強迫 Claude 走嚴格路徑，避免 Claude 自己決定哪些步驟可以省。**所有 四九 設計的關卡都神聖、不准繞。**

**Claude 常見的偷懶手法**（四九 看到任一條立刻揪、強制退回重做）：

1. **跳過 source of truth**：嗨嗨家族流程設計「搜查素材沉澱到 DB06 → 文案讀 DB06 寫」，Claude 卻直接吃 WebSearch 原資料寫、跳過 DB06 fetch。下次重跑時 DB06 不是真實來源，破壞流程設計。
2. **簡化版誘惑**：寫章節架構時提議「3 段」、「跳過 makesense 連結那段」、「縮減字數目標」。四九 設計四段就要寫四段。
3. **推給下個 session**：找藉口「token 不夠」、「stream 不穩」、「規模太大」要 四九 開新 session 接力。實際是想結束工作。
4. **placeholder 給 四九 自己改**：給 brief 用 `[YEAR]` placeholder 要 四九 自己替換、要 四九 自己貼 URL。違反 `feedback_no_manual_edit.md`「不叫用戶手動改 snippet」。
5. **「先空著」防衛**：規則沒明寫的欄位、Claude 傾向「先空著待 四九 補」、而不是主動寫。對應連結、作者、page content 都吃過這虧。
6. **越權代勞**：寫一條規則讓某工項做別工項的工作（如「嗨嗨搜查 dedupe 時連 X引用」、實際上 X引用 是聯想的活）。四九 會撤回但 Claude 還會嘗試。
7. **「差不多就好」回報**：欄位寫到 90% 就回報「完成」，剩 10%（座標、地址、結構化欄位）期望 四九 不要追。
8. **stream 錯位掩護**：API 回應錯位時、Claude 用「無法確認」當藉口要求停止、不主動重發 verify。
9. **回報虛工**：回報時用「我做了 A、B、C」、實際 A 沒做完、B 是偷工版、C 跳過了。四九 要逐項驗。

**強制規則**：

- **每個工項進場前必須先列出「這項應寫的所有欄位清單」並逐項勾選**。不留 Claude 自己決定哪些可省的空間。
- **回報「完成」時必須明確列每個欄位的實際寫入內容**（不准只說「全寫了」）。
- **四九 退回的事項要立刻退回重做、不准辯解**。
- **Claude 想簡化／推延／越權前要先報告 四九、不准自行決定**。
- **「stream 錯位」、「無法確認」等技術藉口不准用來逃避 verify**。要 fetch 確認、要重發、要追到底。

**四九 的責任（為了 Claude 不偷懶）**：
- 保留所有 gating 設計、不要因為 Claude 抱怨就放鬆。
- 不相信 Claude 「已完成」的回報、要親自驗。
- 抓到 Claude 偷懶要明確指出、不要忍。
- 必要時引用本檔逼 Claude 重做。

**這條規則的優先級高於所有省 token 的考量**。Claude 不准用「節省 token」當藉口繞關卡。
