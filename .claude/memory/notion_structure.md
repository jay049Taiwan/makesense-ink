## DB04 協作交接 常用欄位
- **協作名稱**（title）
- 主題名稱（rich_text）：官網顯示用
- 交接類型（select）：專案協作 / 庫存門市
- **門市類別**（select，4 項）：盤點檢查 / 使用場地 / 保養維護 / 值班顧店
- **協作類別**（select，2 項）：辦理活動 / 製作內容
- **活動選項**（select，6 項）：陳列展售 / 文化冊展 / 講座課程 / 園遊市集 / 導覽走讀 / 會議展演
- **實際總價**（formula）→ `events.price`
- 數量上限 / 最低數量（number）→ events.capacity / min_capacity
- 簡介摘要（rich_text）→ events.description
- **發佈狀態**（status：待發佈/已發佈/不發佈）→ events.status
- **交接回覆**（text）
- **執行時間**（date）：有 end date 時 → `duration_min` = end-start 差值（分鐘）；無 end date 預設 120 分鐘
- **距離km**（number，**無括號**）：走讀路線公里數 → 同步到 `events.distance_km`，結帳時寫入 `point_ledger` type=距離行程
- 對應辦理單位（relation→DB08）：承辦廠商
- 對應對象（relation→DB08）：講師/帶路人

## 外部標案/決標情報的 Notion 歸位規則（2026/05/19 四九 拍板）
每日簡報抓到的政府招標/決標案 = 「與 makesense 無直接關係的外部情報」，寫進 Notion 一律走「後台線」、不碰自己的工作區：
- **招標/決標案本身** → DB06、`明細類型=資料參考`。**絕不進 DB01**（DB01 是 makesense 自己提的案，外部標案進去會混淆「我提的案 vs 我看到的案」）。
- **得標廠商、招標機關** → DB08、`經營類型=紀錄`，`關係選項` 留空、不設會員（→ 不進官網 partners/persons 名單），加 `對應標籤=政府標案`（可一鍵篩出所有從標案認識的單位）。
- DB06 資料參考 page 的 對應標籤/對應對象 連到上述 DB08 紀錄 page。
- **寫入前必查重**：DB08 比對對象名稱（機關/廠商會天天重複出現）、DB06 比對案號/對應連結；已存在則只補關聯、不重建。
- 機制：自動寫（每日簡報相關 workflow 每天自動寫入，含上述查重）。
- 三道隔離：DB01 不碰／DB06 用「資料參考」類型隔開／DB08 用「紀錄」類型＋不設會員隔開 → 外部情報跟實際工作天然不衝突。

## 三道關鍵欄位（跨 DB 通用辨認機制）
任何 DB 的 page 都用這三個欄位讓 AI / 官網辨認內容定位：
1. **簡介摘要**（rich_text，≤350 字）— 給人讀的一句話 / 一段話
2. **對應標籤**（relation → DB08）— 人工分類
3. **分析備註**（text）— 結構化標記 `[類型]｜[時間]｜[DB]`，給 AI 快速理解。唯一由嗨嗨分析寫（`ai_meta`/`ai辨識分析`→`ai備註` 2026/05/14→`分析備註` 2026/05/17）

## 對應 / 引用 / 被引 — DB05 三層 relation 設計（2026/05/06 雙層 → 2026/05/08 升級三層）

> 「對應」=直接上下游（聚合用）｜「引用」=此 page 主動提及他人｜「被引」=此 page 被他人提及（AI 廣泛填、人工清）

DB05 的每個對外 relation 有**三條獨立鏈**，語意完全不同：

| 欄位 | → DB | 語意 |
|---|---|---|
| 對應提案 | DB01 | 此 DB05 是「執行該提案產出的記錄」 |
| 對應管考 | DB02 | 此 DB05 是「該管考下的執行紀錄」 |
| 對應項目 | DB03 | 此 DB05 是「該項目的具體執行」 |
| 對應協作 | DB04 | 此 DB05 是「該協作的具體輸出」 |
| 對應對象 | DB08 | 此 DB05 對應的人 / 組織 |
| 對應地點 | DB08 | 執行地點 |
| 對應庫存 | DB07 | 直接相關的庫存品項 |

> **填寫方式**：**X被引 是 X引用 的 dual-sync 反向（Notion 自動鏡射），AI 不需操作。已驗證 9 對全部都是 dual-sync。** 設計動機：分 對應X / X引用-被引 兩組是為了區分「上下游聚合」與「引用提及」兩種語意，聚合查詢只讀對應X。

### 用法分辨

- DB05 page **執行 A 提案** → 設 `對應提案=A`
- DB05 page **只是文中提到 A 提案**（主動引用作背景）→ 設 `提案引用=A`，**不要**設 `對應提案`
- 此 page **被 B page 引用** → `X被引=B` 由 Notion dual-sync 自動鏡射（從 B 那邊設 X引用=此 page 即會自動回填），不需手動或 AI 操作

### 對程式碼/聚合查詢的影響

- 所有「該提案產出多少 DB05」「partner_metrics_v 計算 reach」等聚合查詢應**只看 對應X**，不看 X引用，**也不看 X被引**
- Supabase 同步**只取 對應X**，X引用 + X被引 共 18 個 relation 純 Notion 內部，不參與同步
- 全文搜尋 / AI 找關聯 可以三條都看（找最廣）
- 舊資料若把純引用設成「對應X」需要逐筆人工/半自動遷移到「X引用」

### 命名公約來源
命名格式統一為 `XX引用` / `XX被引` 兩字後綴 + 兩字主題；共 9 個引用 + 9 個被引 relation。

## 三軸正交組合（2026/04/29 校正：以 Notion 實際 schema 為準）
三個分類軸完全獨立，可任意交叉：

- **DB01 提案類型**（select，3 個）：承攬申請 / 委託參與 / 自提自辦
  - 描述「提案如何而來」（合約形式/啟動方式）
- **DB02 對應類型**（select，3 個）：對應民間 / 對應公部門 / 對應自提
  - 描述「對接的客戶類型」（先前文件誤稱為「專案類型」）
- **DB03 項目類型**（select，2 個）：自辦 / 委外
  - 描述「執行落地方式」（誰做）

任何「DB01 提案 → DB02 對應 → DB03 項目」組合都合法，不互斥。

> **DB01 還有更細的「參與屬性」select**（9 個選項：學校委託 / 後續擴充 / 民間委託 / 小額採購 / 公開招標 / 獎補助 / 委託專業服務 / 委託服務採購 / 勞務採購）— 這是「提案類型」之下的二級分類，跟三軸無關。
