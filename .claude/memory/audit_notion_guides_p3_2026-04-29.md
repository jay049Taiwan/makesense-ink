# 1. 官網內容撰寫指南（3329ff25fdab8017b9a3eae1fb7fb5a2）

> 本指南是 P3 三份中問題最重的一份。指南頂部已改寫為 Supabase 新架構，但中段「文章/主題策展」「修改紀錄」與「六、舊版 WP 對照表」仍殘留錯誤的選項名與不存在的選項。

## 🔴 致命

**1.1 「主題策展」全篇仍在用，但 DB05 `官網備項` 已沒有此選項**
- DB05 `官網備項` 實際 select：`['地方通訊','觀點論述','活動企劃','商品介紹','地方調研（footer)','展售合作（footer)','空間體驗（footer)','關於我們（footer)','走讀漫遊（footer)','話題推薦','地方學堂']`
- 指南內仍出現「主題策展」位置：
  - §「文章(post) → 主題策展」段落（篇幅 150–300 字、範例「山林書單｜跟著樹的律動向上走」）
  - §「主題策展」獨立 section（規範 DB05 欄位對應）
  - §「六、舊版 WP 對照表」第 4 條「主題策展 → theme_collection」、n8n 分流流程圖、各路徑必填欄位表
- 4/28 已知改動：「主題展售→話題推薦」。但指南文中所有「主題策展」都未跟著改成「話題推薦」。AI 依此寫作會用一個不存在的 `官網備項` 值。

**1.2 「內容來源與官網對應」表把 `話題推薦` 對應到「products + topics 關聯／主題選書區塊」，但 `官網備項=話題推薦` 的內容來源實際上是 DB05 自己（articles）**
- 表上「Supabase 表」欄寫「透過 products + topics 關聯」，「官網路徑」寫「官網『主題選書』區塊」
- 對照官網維護指南第二節 B2/BS1 的「主題選書」其實是 `products WHERE category LIKE '商品/選書'`，根本不是 DB05 `官網備項=話題推薦`
- 結論：話題推薦的真正去向（topics？articles？或 DB05 獨立列表？）指南裡沒有寫對；建議與官網維護指南交叉確認後重寫此列。

**1.3 §「觀點漫遊」的 DB08 篩選條件寫錯**
- 指南：「DB08 關係經營（經營類型=觀點，經營類型=觀點）→ Supabase topics 表」
- 實際 DB08 `經營類型` 選項 = `['觀點','標籤','紀錄']`；正確篩選是 `經營類型=觀點` 一個就夠（重複兩次是筆誤），且應補上 MEMORY.md 已記錄的「topics tag_type 映射：觀點→viewpoint、標籤→tag、紀錄→不同步」。

**1.4 「官網備項」括號全形 vs 半形不一致**
- DB05 schema 的選項是半形括號：`地方調研（footer)`、`展售合作（footer)` …（左全形「（」、右半形「)」，這是 Notion 既有問題）
- 指南頂部對照表照抄 schema（`地方調研（footer)` 半形右括號），符合
- 但 §機構形象頁撰寫規範 內所有「官網備項=展售合作」「官網備項=空間體驗」「官網備項=地方調研」「官網備項=關於我們」**漏了 `（footer)` 後綴**——AI 直接拿這些字串去寫 Notion 會 select 不到值。

## 🟡 過時

**1.5 §「文書備項」相關文字** — 指南未獨立成節，但 CLAUDE.md / MEMORY.md 對照值得提：DB05 實際 `文書備項`=`['新聞稿','提案申請','成果報告','報價報表','憑證文件','公文']`，CLAUDE.md 的描述（叫「文書類型」、缺「新聞稿」）需修正——但這屬於 CLAUDE.md 的事，不是指南本身的錯。指南本身對 `文書備項` 沒有規範，建議補一節「文案細項=行政文書 → 文書備項」決策樹。

**1.6 §「文案細項」實際選項與指南所述不符**
- DB05 實際：`['論述發想','腳本企劃','訪查調研','簡報圖文','官網內容','網頁社群','行政文書']`
- 指南僅在「地方通訊（文章類內容）」表中列 `文案細項=官網內容`，沒有列其他六項在何種情境下選用。MEMORY.md 對應記錄也是舊版（含「新聞報導/腳本教案/出版品」這些已不存在的選項），指南需補完整 7 項對照表。

**1.7 §「DB05 必填欄位對應」表（路徑 7. 關於我們）的子選項**
- 指南列六個子選項：「品牌簡介、發展歷程-階段、發展歷程-年、發展歷程-季、實踐能力、營運績效」
- DB05 `官網備項` 只有一個 `關於我們（footer)` 選項，**沒有任何子選項欄位**對應這六個區塊。實際應該另有一個欄位（可能是「明細內容」或其他 select）來區分區塊；指南未說清楚這個對應機制——n8n 拿到 `關於我們（footer)` 一筆資料時無法判斷要更新六個區塊的哪個。

**1.8 §「文章(post) → 活動文(300–500 字)」與「活動(event)」段落重複**
- 兩段都規範單場活動的撰寫要點；活動實際來源是 DB04，「活動(event)」一節有完整 DB04 欄位對應，「文章(post) → 活動文」這段純概念，建議刪除以避免雙重來源。

**1.9 §「地方學堂」標記為「Supabase quizzes 表（待建立）」**
- list_tables 確認 Supabase 至今沒有 `quizzes` 表。指南本身正確，但「待建立」狀態到 5/4 沒進展，建議在「待確認事項」清單再標記一次。

**1.10 §「修改紀錄」最後一筆是 2026.03.31，沒紀錄 2026/04/15 Supabase 遷移以外的後續更新（4/28 主題展售→話題推薦），時間軸斷層。**

## 🟢 次要

**1.11** 「官網內容種類」表的「Supabase 表」欄多處寫「待確認」（關於我們/地方調研/展售合作/空間體驗/走讀漫遊）——這些 footer 頁面其實大多在 page_structure_footer_pages.md 已寫清楚是「DB05 官網備項 + Supabase 動態資料」，可以直接補上「articles + 對應 Supabase 表（events / partners / space_bookings / places）」。

**1.12** 範例文字裡的「綯情境」「鬼履有感杖」等是輸入錯字（OCR/輸入法殘留），讀者會分心，建議掃過全文修錯。

**1.13** §「六、舊版 WP 對照表」既然已標「已過時，僅供參照」，建議直接折疊或挪到附錄，避免 AI 撰寫時引用到舊路由。
