# 嗨嗨檢核(hihicheck)

被 `/hihicheck <輸入>` 觸發。任務:對輸入做驗收檢查,**結果只在 session 回覆,不寫任何外部系統**。

---

## 啟動鐵律(全模式通用,不可違反)

1. **唯讀** — 不 PATCH Notion、不執行 n8n workflow、不寫檔案、不 commit。
2. **不信母 session context、抓現場** — 不管母 session 看過/改過什麼,**一律派子 agent** 去 fetch 線上狀態。母 session 的記憶會偏向確認自己的工作,子 agent 沒有 context、是乾淨的眼睛。**這是結構保證,不靠自律。**
3. **每條發現帶證據** — 頁名 + 段落 + 原文片段。不模糊、不省略。
4. **嚴重度標籤** — 大卡關 / 敷衍 / 小問題 三級分。
5. **子 agent 的結論也要驗** — 子 agent 報「找不到」常常是型別/路徑找錯(例:Notion 共享區塊在 toggle 子層,型別是 `synced_block_reference` 不是 `synced_block`)。結論可疑時自己再 fetch 一次。

---

## 輸入分流

### 模式 1:Notion URL
輸入是**純** Notion URL(`notion.so/...` 或 32 碼/dash 格式 page ID),沒夾雜其他實質文字 → **pipeline 檢核**(下方 §模式 1)

### 模式 2:結構化完成報告
輸入是文字,且至少含:
- 修了/做了什麼(逐項列出)
- 每項的位置(Notion page ID、workflow ID、檔案路徑,**至少一項可去查的座標**)

→ **報告驗收**(下方 §模式 2)

### 模式 3:含混 / 空 / 無位置
輸入空、只說「幫我檢查」、只說「我改好了」、或聲明逐項但**沒給任何可去查的位置** → **退件要表單,不開跑**。

回覆:

```
無法檢查 —— 輸入沒給可去查的位置。請補上下面的表單再貼一次:

[上輪要求] 我上次請求你做什麼
[本次回報] 我這次做了什麼(逐項 + 位置:Notion URL / workflow ID / 檔案路徑)
[驗收標準] 怎樣算做對(若有)

任何一項缺位置我都查不到,只能退件。
```

並停止。**絕對不腦補位置、不開跑、不依母 session context 推測**。

## 模式 1:Pipeline 檢核(Notion URL)

對 target page 跑 **4-2-7-1 嗨嗨檢核 §三 5 項清單檢核**:

| 項 | 應寫欄位 | 檢查 |
|---|---|---|
| 1. 企劃 | `執行構想`(非空 ≥3 句);後期 `主題名稱`、`簡介摘要`(60-100 字) | 達標 |
| 2. 搜查 | 9 個 X引用 至少 1 個有值 | 連到 DB06 page 的 `明細類型=資料參考` |
| 3. 分析 | X引用 連到的所有 DB06 都有 `對應標籤` | 每筆 ≥1 個 DB08 |
| 4. 聯想 | X引用 ≥3 筆,跨 ≥2 個 DB | 量 + 多樣性 |
| 5. 文案 | target `page content`、`發佈狀態=待發佈`;page content 跟 `執行構想` 語意對應 | 文案是否完成構想交辦 |

### 執行(派 1 個子 agent)

派 `general-purpose` 子 agent。**Prompt 範本(直接複製,把 `<TARGET_URL>` 換掉)**:

```
你是嗨嗨檢核的子 agent。READ-ONLY,不修改任何 Notion 內容。

目標:對下面這個 Notion target page 跑 5 項 pipeline 檢核。

Target: <TARGET_URL>

Notion MCP fetch 工具是 deferred。先呼叫 ToolSearch query="select:mcp__73449d04-01f2-4d79-b0b5-908d1d64b42d__fetch" 載入。

頁面可能很大;若 fetch 回傳 saved file path,用 python 字元切片分段讀完整(python3 -c "s=open('path').read(); print(s[0:18000])")。

讀完後,逐項判定:
1. 企劃:執行構想 非空 ≥3 句?後期 主題名稱、簡介摘要(60-100 字)?
2. 搜查:9 個 X引用 至少 1 個有值,連到的 DB06 page 是否 明細類型=資料參考?
3. 分析:X引用 連到的每筆 DB06 是否都有 對應標籤(≥1 個 DB08)?
4. 聯想:X引用 是否 ≥3 筆且跨 ≥2 個 DB?
5. 文案:target page content 是否存在?發佈狀態=待發佈?page content 是否在語意上回應 執行構想 交辦的事?

每項判 ✅/⚠️/❌,附證據(實際讀到的欄位值片段)。
最後寫一段「語意對應」總結:文案有沒有完成構想交辦?符合/有偏離說明。
報告以繁體中文回。
```

### 主 session 整合輸出

```
🔍 嗨嗨檢核(Pipeline 模式)
對象:[target 名稱](URL)
執行構想:[摘要 1-2 行]

| 項 | 應寫欄位 | 結果 | 證據 |
|---|---|---|---|
| 1. 企劃 | 執行構想 / 主題名稱 / 簡介摘要 | ✅/⚠️/❌ | … |
| 2. 搜查 | X引用 + 資料參考 | … | … |
| 3. 分析 | 對應標籤 | … | … |
| 4. 聯想 | X引用 ≥3 / 跨 2 DB | … | … |
| 5. 文案 | page content + 發佈狀態 + 語意 | … | … |

語意對應:[符合 / 偏離說明]
建議:[補某項 / 或標完成]
```

## 模式 2:報告驗收(結構化完成報告)

### 解析

從輸入抽出:
- **聲明清單**:每條做了什麼(R1, R2, ... 或 #1, #2, ... 或任何條列)
- **每項位置**:Notion page ID、workflow ID、檔案路徑(至少一項)
- **驗收標準**(若報告有寫):怎樣算做對

**任何一項沒位置 → 退件問,不腦補**。

### 派工(並行多個子 agent)

按位置類型分群,**並行**派多個 `general-purpose` 子 agent:
- 屬 Notion 的項 → 一個 Notion 子 agent
- 屬 n8n 的項 → 一個 n8n 子 agent
- 屬檔案的項 → 一個檔案子 agent

**Notion 子 agent prompt 範本**:

```
你是嗨嗨檢核的 Notion 子 agent。READ-ONLY,不改任何頁面。

任務:驗證下列聲明是否屬實。逐條判 ✅/⚠️/❌ + 證據(原文片段)。

聲明清單:
[項目 1] [位置:Notion URL/ID] [宣稱做了什麼]
[項目 2] ...

Notion MCP fetch tool 是 deferred。先 ToolSearch query="select:mcp__73449d04-01f2-4d79-b0b5-908d1d64b42d__fetch" 載入。

頁面可能很大;fetch 若回傳 saved file path,用 python 字元切片分段讀(python3 -c "s=open('path').read(); print(s[0:18000])")完整讀完。

【務必抓的反模式(逐條檢查每一項)】
1. 頁尾貼警語、正文不動 — 對比頁尾說的 vs 正文寫的
2. 同頁正文 vs 頁首鐵律 互打 — 全頁掃,找矛盾
3. 修一個漏兩個 — 同款問題只清部分實例(看完整集合,不只看主張的那個)
4. Synced block 沒清 — Notion 找型別 `synced_block_reference`(不是 `synced_block`),進 toggle/details 遞迴往下找
5. 命名前後不一 — 同概念用兩個名(比對欄位字典)
6. 內部互打不管大小 — 小矛盾不可放過,一律標

【對任何「找不到」結論要懷疑】若一條判定「沒這東西」之前,
換 1-2 種搜尋型別/路徑再驗一次(尤其 synced block 是常見陷阱)。

報告以繁體中文回。逐項 ✅/⚠️/❌ + 證據片段。
```

**n8n 子 agent prompt 範本**:

```
你是嗨嗨檢核的 n8n 子 agent。READ-ONLY,不執行/修改任何 workflow。

任務:驗證下列聲明。逐條判 ✅/⚠️/❌ + 證據。

聲明清單:
[項目 1] [workflow ID] [宣稱做了什麼,例:已封存 / 已移除硬編碼 key]
[項目 2] ...

n8n MCP 工具是 deferred。先 ToolSearch query="select:mcp__a7fc8d57-4921-4104-b7bd-d05f8990caea__get_workflow_details,mcp__a7fc8d57-4921-4104-b7bd-d05f8990caea__search_workflows" 載入。

對每條:call get_workflow_details,檢查實際狀態(active/archived、node code 內容、是否還有硬編碼 secret 等)。

【務必抓的反模式】
1. 修一個漏兩個 — 同款問題只清部分實例(尤其硬編碼 token 常見漏網)
2. archive ≠ delete — 確認到底是封存還是真關閉
3. 對方沒提到的同款問題 — 主動掃一遍(例:聲明 WF1b 已清,順手檢查還有沒有別的 workflow 同樣硬編碼)

報告以繁體中文回。逐項 ✅/⚠️/❌ + 證據。
```

**檔案子 agent prompt 範本**:

```
你是嗨嗨檢核的檔案子 agent。READ-ONLY,不修改任何檔案。

任務:用 Read 工具讀下列檔案,驗證聲明。

聲明清單:
[項目 1] [檔案路徑] [宣稱做了什麼]
...

對每條:Read 該檔,判 ✅/⚠️/❌ + 證據(實際讀到的片段)。

報告以繁體中文回。
```

### 主 session 整合輸出

```
🔍 嗨嗨檢核(報告驗收模式)
驗收對象:[對方宣稱「X/Y 全部結案」]

| # | 項目 | 結果 | 證據 |
|---|---|---|---|
| R1 | … | ✅/⚠️/❌ | … |
| R2 | … | … | … |
...

【結論】
- 屬實:N 項
- 表面/未做完:M 項(列出哪些)
- 做錯:K 項(列出哪些)

【退件建議】(若有 ⚠️/❌)
[每條:該怎麼真正修。原則:改正文、不貼警語、清根。]
```

## 反模式速查表(任何模式都要對照)

| 反模式 | 長相 | 抓法 |
|---|---|---|
| 頁尾警語 | 頁尾貼修正聲明,正文沒改 | 對比頁尾 vs 正文 |
| 內部互打 | 同頁兩段講相反的事 | 全頁掃矛盾 |
| 修一漏多 | 同款問題只清部分實例 | 全集合掃,不只看主張那個 |
| Synced block 沒清 | 聲稱刪了但藏在 toggle 子層 | Notion 找 `synced_block_reference`,進 toggle 遞迴 |
| 命名前後不一 | 同概念用兩個名(庫存選項 vs 庫存細項) | 比對欄位字典 |
| 小矛盾放過 | 「太小不用修」 | **一律標**,沒有「太小」 |

---

## 結尾鐵律

- **任何子 agent 結論可疑** → 自己再 fetch 一次確認,不照單全收
- **發現任何反模式** → 即使對方主張全部結案,仍列出來,不要客氣
- **絕對不寫外部系統** — 報告只回 session 內。要正式紀錄寫回 Notion,請走 n8n 檢核棒 v5 workflow,不是這條手動通道
- **輸入無位置 → 退件**,不腦補、不依母 session context 推測
