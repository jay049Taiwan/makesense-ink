# /explain — 白話解說技能

## 核心原則

1. **從業務影響出發**：不從技術原理出發，從「這對現思的工作流程有什麼差別」出發
2. **具體例子優先**：抽象概念一定要接上現思的實際業務（DB06 庫存、DB04 活動、嗨嗨家族等）
3. **技術詞一定加中文**：embedding（嵌入/向量化）、webhook（鉤子/觸發器）、表達為中文括號說明
4. **解說長度適配**：一個名詞 → 2-3 句；一段機制 → 條列 + 一個具體流程圖；複雜架構 → 分層拆解
5. **不賣弄**：能用一句話說清楚的，不用五句話

## 常用解說模板

### 名詞解說格式
```
[名詞] 是什麼：
一句話定義。

在現思的用途：
具體怎麼用，影響哪個工作流程。

打個比方：
用日常生活或現思業務的比喻。
```

### 機制解說格式
```
[機制名稱] 的運作流程：

觸發條件 → 步驟一 → 步驟二 → 結果

舉例：
在現思的實際情境下，這個機制解決了什麼問題。
```

## 現思系統常用解說備查

### 嗨嗨家族分工
- **嗨嗨總管**：接任務、分派、問狀況的管理者
- **搜查棒**（search）：去外部資料源（網路、Notion）找資料
- **分析棒**（analyze）：消化原始資料，產出摘要和見解
- **企劃棒**（plan）：從分析結果生出行動方案
- **文案棒**（copy）：把企劃變成可發佈的文字
- **檢核棒**（check）：確認輸出品質、格式、一致性

### 技術詞白話
| 技術詞 | 白話說法 |
|--------|---------|
| webhook | 「按一個按鈕，自動通知另一個系統」的機制 |
| embedding | 把文字變成數字向量，讓電腦可以算相似度 |
| upsert | 有就更新、沒有就新增（省得先查再決定） |
| PATCH | 局部更新（只改你指定的欄位，其他不動） |
| cron job | 定時自動執行的排程任務 |
| errorWorkflow | n8n 專用的「出錯時去找哪個 WF 報警」設定 |
| continueOnFail | 這個節點出錯也不停下來，繼續跑下一節點 |
| pgvector | PostgreSQL（Supabase）的向量搜尋擴充套件 |
| HNSW index | 向量搜尋的加速索引（近似最近鄰搜尋） |

## 使用方式

```
/explain embedding
/explain 嗨嗨派發中心的運作
/explain dispatch WF 裡 continueOnFail 的作用
```

解說時先確認對象是什麼層次的問題（名詞定義 / 機制流程 / 架構設計），再選對應的模板輸出。
